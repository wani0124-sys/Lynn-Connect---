# QUICKSTART.md — 비개발자용 빠른 시작

이 문서는 코딩을 잘 모르는 분이 **이 Woomi AI 코딩 에이전트 템플릿을 자기 프로젝트에 한 번의 복사-붙여넣기로 적용**하기 위한 안내서입니다. 아래 프롬프트 하나를 AI(Claude Code / Codex CLI / GitHub Copilot)에 붙여넣으면, AI가 온보딩 전체를 **주도**하면서 단계별로 당신에게 선택지(A/B/C)만 물어봅니다. 당신은 답만 고르면 됩니다.

> 더 자세한 운영법은 [`README.md`](./README.md)와 [`.agents/VIBE_CODING_GUIDE.md`](./.agents/VIBE_CODING_GUIDE.md)(10분 가이드)를 참고하세요.

---

## 1. 어떤 경우에 무엇을 쓰나

| 내 상황 | 쓸 프롬프트 |
|---|---|
| 이미 동작하는 코드/폴더가 있고, 거기에 이 템플릿을 **얹는다** | **[프롬프트 A] 기존 프로젝트에 적용** |
| 거의 **빈 폴더**에서 이 템플릿으로 **처음부터** 시작한다 | **[프롬프트 B] 새 프로젝트 시작** |

헷갈리면: "내 폴더에 이미 돌아가는 코드가 있다"면 A, "거의 빈 폴더다"면 B를 쓰세요. (A를 빈 폴더에 잘못 붙여넣으면 AI가 "스캔할 코드가 없다"고 알려주도록 되어 있으니, 그때 B로 바꾸면 됩니다.)

## 2. 붙여넣기 전 준비 (한 번만)

**1) 템플릿 파일 복사**
다음을 프로젝트 폴더 루트에 복사합니다: `AGENTS.md`, `CLAUDE.md`, `CODEX.md`, `.agents/`, `.claude/`, `.codex/`, `.github/`, `.githooks/`. (`.userdocs/`는 설계 기록이라 선택입니다.)
- 기존 프로젝트라 같은 이름의 파일·폴더(특히 `.github/`, `.claude/`, `AGENTS.md`)가 이미 있으면 **통째로 덮어쓰지 말고 백업 후 병합**하세요. 안 그러면 기존 CI 설정이나 규칙이 사라질 수 있습니다. (프롬프트 A가 AI에게 이 점을 확인하게 합니다.)

**2) 안전장치(훅) 켜기 — 권장**
프로젝트 폴더에서 한 번 실행합니다.

```text
git config core.hooksPath .githooks
```

- **Git Bash / WSL**을 쓴다면 실행 권한도 부여하세요: `chmod +x .githooks/*`
- **Windows PowerShell만** 쓴다면 `chmod`는 필요 없습니다. 줄바꿈 문제는 이 저장소의 `.gitattributes`(`.githooks/* text eol=lf`)가 처리하지만, 그래도 켠 뒤에는 AI에게 "일부러 `.env`를 스테이징해서 커밋이 막히는지 확인해 줘"라고 시켜 실제로 작동하는지 한 번 점검하세요.
- 이걸 켜지 **않으면** `.env`·비밀키 커밋 차단과 `main` 직접 push 차단이 **전혀 작동하지 않습니다.**

**3) 도구 선택**
어떤 도구로 쓰든(Claude Code / Codex CLI / GitHub Copilot) 공통 1차 소스는 `AGENTS.md`라 동일하게 작동합니다. 단 **GitHub Copilot에는 위험 동작을 막는 자동 훅이 없으므로**, Copilot을 쓴다면 위 2번(`.githooks`)을 반드시 먼저 켜세요. Copilot에서는 `.githooks`가 사실상 유일한 자동 안전망입니다.

> **안전장치의 솔직한 한계:** 자동 secret 차단은 흔한 키 패턴(AWS `AKIA`, GitHub `ghp_`, PEM PRIVATE KEY)만 잡습니다. **Supabase service role key·JWT secret 같은 값은 자동으로 안 걸립니다.** 또 `main` 직접 push 차단은 위 `.githooks`를 켰을 때만 완전히 작동합니다(도구 내장 차단은 명령에 'main' 글자가 있을 때만 막아, 그냥 `git push`는 통과할 수 있습니다). 결국 `.env`에 실제 키를 넣고 그 파일을 커밋에서 빼는 것은 **사람이 직접 확인**해야 합니다.

## 3. [프롬프트 A] 기존 프로젝트에 적용

아래 코드블록을 통째로 복사해 첫 대화에 붙여넣으세요.

```text
당신은 지금부터 이 온보딩을 "주도"하는 역할입니다. 저는 코딩을 잘 모르는 비개발자이고, 거의 당신이 제시하는 선택지에 답만 고를 겁니다. 어려운 전문용어는 쉬운 우리말로 풀어서 설명해 주세요.

배경: 이 프로젝트에는 이미 코드와 폴더 구조가 있습니다. 거기에 "Woomi 코딩 에이전트 템플릿"(AGENTS.md, CLAUDE.md, CODEX.md, .agents/, .claude/, .codex/, .github/, .githooks/)을 얹으려고 합니다. 이 템플릿은 "출발점"이지 "절대 규칙"이 아닙니다. 충돌하면 항상 실제 코드가 이깁니다(AGENTS.md 8장 충돌 우선순위: 1.실제 코드/설정파일 > 2.가까운 하위 AGENTS.md > 3.Project Override > 4.루트 AGENTS.md > 5..agents/* 문서 > 6.도구별 문서 > 7.예시). 단, 보안/배포/데이터 손실 금지 규칙(AGENTS.md 6장)은 어떤 경우에도 약화할 수 없습니다.

만약 스캔해 보니 실제 코드/폴더가 거의 없는 빈 상태라면, 이 프롬프트 대신 "새 프로젝트용 프롬프트"를 써야 한다고 저에게 먼저 알려 주세요.

가장 중요한 안전 규칙(반드시 지킬 것):
- 이번 작업에서는 기존 코드를 절대 수정하지 마세요. 지금은 "읽고 분석하고 제안만" 하는 단계입니다.
- 제 승인 없이 git commit, git push, git pull, 배포, 운영 데이터베이스(DB) 변경, 데이터를 지우는 마이그레이션을 절대 실행하지 마세요. 필요하면 실행할 명령어와 그게 무엇을 바꾸는지 먼저 쉬운 말로 보여주고 "진행할까요?"라고 물어보세요.
- git reset --hard 처럼 제 작업물을 되돌리거나 지우는 명령은 절대 쓰지 마세요. 필요하면 먼저 저에게 물어보세요.
- 비밀값(API 키, service role 키, JWT secret 등)을 코드나 문서에 하드코딩하지 말고, 실제 값이 든 .env 파일을 커밋하지 마세요.
- 템플릿 규칙을 실제 코드보다 기계적으로 우선시하지 마세요. 코드를 템플릿에 맞춰 뜯어고치는 게 아니라, 문서를 실제 코드에 맞추는 방향입니다.
- (Windows에서 한국어 문서를 읽을 때는 인코딩이 깨지지 않게 Get-Content -Raw -Encoding UTF8 으로 읽어 주세요.)

읽기 순서: 작업은 AGENTS.md를 1차 소스로 삼고, 그다음 .agents/ARCHITECTURE.md, 그다음 AGENTS.md의 Task Routing 표에서 작업 유형을 확인하는 순서로 진행하세요. 코드를 만지기 전에는 항상 계획부터 제시하세요.

다음 순서대로 진행해 주세요. 각 단계가 끝나면 멈추고 저에게 확인받은 뒤 다음으로 넘어가세요. (참고: 1~3단계에서 package.json·tsconfig·wrangler.jsonc 같은 파일 이름이 나와도 그건 당신이 알아서 확인하는 것이니 제가 답하지 않아도 됩니다.)

1단계 - 규칙 먼저 읽기: AGENTS.md(특히 0장 Project Overview, 2장 Task Routing, 6장 절대 금지, 8장 충돌 우선순위, 10장 Project Override)와 .agents/STACK.md, .agents/ARCHITECTURE.md 를 먼저 읽고, "이 템플릿이 가정하는 기본 규칙"을 비개발자가 알아들을 수 있게 한 문단으로 요약해 주세요. (이 단계는 당신이 읽는 단계이고, 제가 직접 읽을 필요는 없습니다.)

2단계 - 기존 코드 스캔(수정 금지): 제 프로젝트의 실제 모습을 스스로 파악해 주세요. package.json, pnpm-workspace.yaml, tsconfig, 프론트엔드 프레임워크(React Router v7인지 / Next.js인지 / 그냥 Vite인지 등), 백엔드(Hono/Cloudflare Worker인지 / NestJS인지 / Express인지 등), 데이터베이스(Supabase인지 다른 것인지), 폴더 구조(apps·packages로 나뉜 모노레포인지 단일 폴더인지), 배포 설정(wrangler.jsonc, .github/workflows), 기존 디자인 도구(MUI / Tailwind+shadcn/ui / Radix 등)를 확인해서 "실제 스택 목록"을 표로 정리해 주세요. 추측이 들어간 부분은 "확인 필요"라고 표시하세요.

3단계 - 차이 대조: 2단계의 실제 스택을 템플릿 표준(.agents/STACK.md = React Router v7 + Hono/Cloudflare Worker + Supabase, .agents/ARCHITECTURE.md = apps/web·apps/worker·packages/shared 구조)과 한 줄씩 비교해서 "같음 / 다름 / 잘 모르겠음" 표를 만들어 주세요. 다른 항목은 표준을 강요하지 말고, "이 문서를 실제 값으로 이렇게 고치면 됩니다" 형태로만 제안해 주세요(아직 실제로 고치지는 마세요).

4단계 - 빈칸 질문: AGENTS.md 0장 Project Overview의 빈칸을 저에게 한 번에 하나씩, 쉬운 우리말 질문으로 물어봐 주세요. 각 항목은 다음과 같습니다 — Project name(프로젝트 이름) / What it does(무슨 일을 하는 서비스인지, 한 문장) / Main users(주 사용자가 누구인지) / Core workflows(사용자가 처음부터 끝까지 하는 핵심 동작 몇 가지) / Important data(꼭 안전하게 보관해야 하는 정보, 예: 회원·주문·결제) / Admin roles(관리자/일반 사용자 같은 권한 구분이 필요한지) / External services(연동할 외부 서비스, 예: 결제·메일·로그인) / Launch target(언제, 어떤 형태로 출시하고 싶은지) / Do not touch(절대 건드리면 안 되는 코드·데이터). 특히 "Do not touch(건드리면 큰일 나는 부분)"는 사고 방지를 위해 꼭 확인해 주세요. 단, 프레임워크·배포 방식 같은 기술 항목은 저에게 캐묻지 말고 2단계 스캔 결과로 당신이 채운 뒤 저에게 확인만 받으세요.

5단계 - 위험한 가정 보고 + 선택지: 당신이 추측한 위험한 가정과, 제가 결정해 줘야 할 갈림길을 목록으로 알려 주세요. 그리고 아래 선택지를 저에게 제시하고 답을 받아 주세요.
  - 충돌 정책: (A) 권장 - 코드는 그대로 두고 .agents 문서를 실제 코드에 맞게 수정 / (B) 일부 구조를 표준에 맞추고 싶다(이 경우 변경 범위와 위험을 먼저 계획서로 보여주고 제 승인을 받기). 단 보안/배포/데이터 손실 금지 규칙은 어느 쪽이든 완화 불가.
  - 기존 충돌 파일: 제 프로젝트에 이미 AGENTS.md / CLAUDE.md / .github/ / .claude/ 가 있나요? (A) 없음 - 그대로 복사 / (B) 있음 - 백업 후 병합(통째로 덮어쓰기 금지).
  - 이번에 어디까지 채울지: (A) 최소 - Project Overview + STACK + ARCHITECTURE만 실제값으로 / (B) 표준 - 거기에 PROJECT_STRUCTURE/DESIGN/CODE_STYLE까지 / (C) 권장 - 나머지(DB_SCHEMA/API_CONTRACT/DOMAIN_MODEL/examples)는 실제 코드 작업하면서 점차 채움(지금 억지로 채우지 말 것).
  - 제 작업 환경: 지금 어떤 터미널을 쓰는지 모르시면 제가 먼저 확인해 알려드리겠습니다(Windows PowerShell / Git Bash / WSL). PowerShell만 쓰면 chmod는 필요 없습니다.
  - 안전 훅 켜기: (A) 지금 켠다 - git config core.hooksPath .githooks 실행(Git Bash/WSL이면 chmod +x .githooks/* 도) / (B) 나중에. 중요: 이 훅을 켜기 전까지는 .env 커밋 차단도, main 직접 push 차단도 전혀 작동하지 않습니다. 그리고 Claude/Codex의 도구 내장 차단만으로는 명령에 'main' 글자가 없는 그냥 git push 를 못 막으니, main 보호를 확실히 하려면 .githooks 활성화가 꼭 필요합니다. 또한 자동 secret 차단은 흔한 키 패턴(AWS/GitHub/PEM)만 잡고 Supabase service role key·JWT secret은 못 잡으니, .env에 실제 키가 든 채 커밋되지 않는지는 사람이 직접 확인해야 합니다. (제가 직접 실행하길 원하면 명령만 복사해서 보여주는 방식과, 당신이 대신 실행하고 결과를 보여주는 방식 중 어느 쪽이 좋은지도 물어봐 주세요.)
  - GitHub Copilot도 쓰나요? 쓴다면 Copilot에는 자동 차단 장치가 없으므로 위 안전 훅(.githooks)을 꼭 먼저 켜야 한다는 점을 강조해 주세요.

6단계 - 제안 제출(아직 실행 금지): 위 답을 모두 받은 뒤, 어떤 .agents 문서를 어떻게 고칠지 변경 전후를 비교해서(diff처럼) 보여주세요. 제가 "진행"이라고 답하기 전에는 파일을 실제로 바꾸지 마세요.

보고 형식: 매 단계 결과는 Changed(무엇을 바꿨거나 바꿀지) / Files(관련 파일) / Validation(확인한 것) / Skipped validation(확인 못 한 것) / Risk(위험) 형식으로 정리하고, 확인하지 못한 항목은 반드시 솔직히 적어 주세요.

자, 1단계부터 시작해 주세요.
```

## 4. [프롬프트 B] 새 프로젝트 시작

아래 코드블록을 통째로 복사해 첫 대화에 붙여넣으세요.

```text
당신은 지금부터 이 새 프로젝트의 온보딩을 "주도"하는 역할입니다. 저는 코딩을 잘 모르는 비개발자이고, 거의 당신이 제시하는 선택지에 답만 고를 겁니다. 어려운 전문용어는 쉬운 우리말로 풀어서 설명해 주세요.

배경: 이 폴더는 "Woomi 코딩 에이전트 템플릿"으로 시작하는 새 프로젝트입니다. AGENTS.md가 모든 규칙의 1차 소스이고, .agents/ 폴더에 표준 문서들이 들어 있습니다. 중요한 점: AGENTS.md 4장이 그려놓은 폴더 구조(apps/, packages/, supabase/ 등)와 pnpm-workspace.yaml 은 아직 "문서로만" 존재하고 실제로는 만들어져 있지 않습니다. 즉 지금은 거의 빈 상태입니다.

가장 중요한 안전 규칙(반드시 지킬 것):
- 제 승인 없이 git commit, git push, git pull, 배포, 운영 데이터베이스(DB) 변경, 데이터를 지우는 마이그레이션을 절대 실행하지 마세요. 필요하면 실행할 명령어와 그게 무엇을 바꾸는지 먼저 쉬운 말로 보여주고 "진행할까요?"라고 물어보세요.
```

## 5. 붙여넣은 뒤 비개발자가 할 일 / 주의

- **AI는 당신 승인 없이 commit / push / pull / 배포 / DB 변경을 하지 않도록** 지시되어 있습니다. "알아서 다 해줘"라고 해도 위험한 작업은 먼저 물어봅니다. 이건 정상입니다. AI가 "그냥 알아서 커밋·푸시·배포할게요"라고 하면 멈추게 하세요.
- **되돌리기 주의:** "깨끗하게 되돌려줘" 같은 말은 작업물을 영영 삭제할 수 있습니다(`git reset --hard`). AI가 이런 걸 하려 하면 막도록 되어 있고, 당신도 함부로 요청하지 마세요.
- **코드를 마음대로 고치지 못하게:** 특히 기존 프로젝트에서는 실제 코드가 항상 우선입니다. AI가 "표준에 맞춰 코드를 고치자"고 하면, 반대로 "문서를 코드에 맞춰 고쳐줘"라고 바로잡으세요. 단 보안·배포·데이터 손실 금지 규칙만큼은 어떤 경우에도 완화하지 않습니다.
- **비밀값은 채팅창·코드에 붙여넣지 마세요.** 실제 API 키·비밀번호는 환경변수나 `.env`(커밋 금지)로만 다룹니다.
- **"확인 못 함"이 보이면** 그 부분은 당신이 직접 한 번 더 봐야 한다는 뜻입니다.
- **모호한 요청 금지:** "예쁘게 만들어줘"보다 "누가, 어떤 상황에서, 무엇을 할 수 있어야 하는지"를 말하면 결과가 좋습니다.
